| 项 | 内容 |

---

## 1. 一句话定位（TL;DR）

把多公司老板 OS（一套纯 Markdown 文件夹）包成一个**桌面应用**：老板把会议语音、报表截图、邮件**拖进去**，后台 AI 自动判断归属公司、归档、更新跨公司汇总，并每天推一条全局摘要。**老板全程不碰终端、不看文件路径**——体验像 Dropbox，区别是这个 Dropbox 会自己整理。

---

## 2. 背景与问题

- 现有 `boss-os-template` 后端已成立：三层信任（Personal / 跨公司汇总 / 各公司）、公司间不串味、会累积的都是文件夹（live+archive）、8 个通用 SOP。
- **但它的入口是开发者向的**：要 clone repo、开 Claude Code、懂目录结构。99% 的老板不会碰终端。
- 真正的瓶颈不在后端，在**入口与呈现**。要让更多老板用，必须把老板和文件系统彻底隔开。

**结论**：不改后端，只补一个"壳 + 后台 agent"，把交互降到"拖拽 + 看仪表盘 + 收通知"。

---

## 3. 目标与非目标

### 目标
1. 老板**零学习成本**：装 app → 拖文件 → 看板/收通知，全程不接触 Markdown/终端。
2. **后端零改动**：数据仍是 `{boss}-os/` 的纯 Markdown，护城河（隔离、私有、可迁移）保留。
3. **快速给一个老板开通**：对话式向导 < 5 分钟生成骨架（复用 `init-boss-os` SOP）。
4. **数据私有**：默认数据留在老板本机 + 老板自己的同步盘，不进我们的服务器。

### 非目标（明确不做）
- ❌ 不自研同步引擎（白嫖 iCloud / Dropbox / 私有 Git）。
- ❌ 不做多租户 SaaS 托管（那是后期，不在本期）。
- ❌ 不做移动端原生 app（手机投递是 P2，靠"分享到同步文件夹"先凑）。
- ❌ 不让老板手动编辑文件结构（所有写操作由 AI 代劳）。

---

## 4. 目标用户

| 角色 | 画像 | 怎么用 |
|------|------|--------|
| **老板（主）** | 同时管 2–6 家公司，时间碎、不懂技术、活在微信/邮件里 | 拖文件进 app、看仪表盘、收每日摘要 |
| **操作员/助理（次）** | 老板的 EA 或你（代运营） | 替老板投递、核对 AI 整理结果、处理边界 case |

---

## 5. 核心概念

> **"It's Dropbox, but it files itself."**

- Dropbox 的魔力 = 一个永远自动同步的文件夹 + 隐形后台。
- Boss OS app 的魔力 = 一个永远自动同步的文件夹 + **一个永远在整理的 AI 后台**。
- 老板看到的是**仪表盘**（今日日程 / 待拍板 / 跨公司现金流 / 各公司卡片），不是 `.md`。

---

## 6. 关键用户场景（User Stories）

1. **拖进来就归档**：老板开完会，把微信导出的语音/纪要拖进 app → AI 转写、判断这是 A 公司的会、在 `Companies/A/Meetings/` 建记录、拆出决策进 `Decisions/`、任务进 `Tasks/`、弹通知"A 公司周会已整理：2 个决策待你拍板"。
2. **财务一拖就更新**：拖一张报表截图 → AI 抽数（GST/ABN 走 `invoice-extract-au`）、更新该公司 `Finance/snapshot.md` 和 `Cashflow/latest.md` → 通知"A 公司现金已更新，跑道 4 个月"。
3. **早晨一条摘要**：每天 08:00 桌面通知/卡片："今天 3 个会、2 件待你拍板、B 公司 BP 今天 deadline"。
4. **扫一眼全局**：老板点开 app 首屏，一屏看完所有公司状态，不用翻文件。
5. **开新公司**：点"+ 公司"，对话式问 5 句 → 自动 `add-company`。

## 7. 功能需求与优先级

### P0 — MVP（一个老板能真用的最小集）
| # | 功能 | 说明 |
|---|------|------|
| F1 | **拖拽投递 + 自动归档** | 监听 Inbox/拖拽区，触发 SOP：meeting-notes / invoice / inbox-triage |
| F2 | **只读仪表盘** | 一屏渲染 Agenda + Tasks + Cashflow + 各公司卡片 |
| F3 | **每日摘要推送** | 定时生成 `Weekly`/今日摘要 → 桌面通知 |
| F4 | **归属确认** | AI 不确定归哪家公司时，弹一个轻确认（防串味铁律 #1） |
| F5 | **同步（外挂）** | 数据放 iCloud/Dropbox 文件夹，跨设备自然同步 |

### P1 — 让它好用
| # | 功能 | 说明 |
|---|------|------|
| F6 | 对话式开通向导 | 复用 `init-boss-os`，< 5 分钟建好一个老板 |
| F7 | 语音投递 | 拖入音频自动转写（接 transcribe） |
| F8 | 待办/决策可点 | 仪表盘点一下标记完成/已拍板，触发归档滚动 |
| F9 | 全局搜索 | 跨公司搜会议/任务/决策（只读） |

### P2 — 规模化
| # | 功能 | 说明 |
|---|------|------|
| F10 | 手机投递 | iOS/安卓"分享到"→ 同步文件夹；或微信机器人转发 |
| F11 | 多老板/代运营台 | 操作员一个界面管多个老板实例 |
| F12 | 主动洞察 | 跑道告警、逾期任务、决策超时自动提醒 |

## 8. 系统架构

```
┌─────────────────────────── 桌面 App（壳） ───────────────────────────┐
│  React 仪表盘  ── 渲染 ──►  {boss}-os/ 的 Markdown（只读视图）         │
│       ▲                                                               │
│       │ 通知                                                          │
│  后台 Agent（常驻）                                                    │
│   - chokidar 监听 Inbox/ 与拖拽区                                      │
│   - 新文件 → 调 Claude Agent SDK 跑对应 SOP                            │
│   - 写回 OS（归档/更新汇总）                                            │
│   - 定时任务：每日摘要                                                  │
└───────────────────────────────┬──────────────────────────────────────┘
                                 │ 读写
                       {boss}-os/  （纯 Markdown，本地）
                                 │
                      iCloud / Dropbox / 私有 Git  （同步层，外挂，不自研）
```

### 技术选型（贴 JR 现有 JS/TS 栈）
| 部分 | 首选 | 备选 | 取舍 |
|------|------|------|------|
| 壳 | **Tauri** | Electron | Tauri 轻、像 Dropbox 菜单栏；Electron 上手更快（团队 React 熟） |
| 界面 | React + Vite | — | 与 JR 现有前端一致 |
| 后台 agent | **Claude Agent SDK (TS)** | 自建调用 NestJS | 直接跑 SOP，本地优先 |
| 文件监听 | chokidar | — | 监听 Inbox/ |
| 转写 | 本地/云 transcribe | — | 语音投递用 |
| 同步 | iCloud / Dropbox / Git | — | **不自研** |

> **取舍建议**：MVP 用 Electron 抢速度（团队 React 直接复用），验证后若要"轻量常驻菜单栏"体验再评估迁 Tauri。

## 9. 关键流程：投递 → 整理 → 通知

1. 文件落入 `Inbox/`（拖拽或同步进来）。
2. 后台 agent 触发，识别类型 → 选 SOP（meeting-notes / invoice / triage）。
3. 判断归属公司；**不确定 → 弹确认**（绝不猜，铁律 #1）。
4. AI 执行 SOP：原始件保留（铁律 #2）、distilled 写结构化区、更新 Agenda/Tasks/Decisions/Cashflow。
5. 原始件移 `Inbox/_done/`。
6. 仪表盘刷新 + 桌面通知。

---

## 10. 隐私与安全（护城河）

- **数据主权**：默认全部留老板本机 + 老板自己的同步盘，**不进我们服务器**。卖点一句话："你的钱和机密在你自己的电脑里，像 Dropbox。"
- **公司间不串味**：跨公司汇总只存引用 + 聚合数字，AI 处理 A 公司时不带入 B 公司上下文（铁律 #1）。
- **Finance 最高密级**：快照+变更记录、不覆盖历史、绝不外发（铁律 #3）。
- **对外动作先确认**：代发邮件等需老板点头（铁律 #4）。
- LLM 调用：敏感数据可选走企业/私有部署端点（后期）。

---

## 11. 非功能需求

- **离线可用**：仪表盘只读渲染不依赖网络；AI 整理需联网（可排队，联网后补跑）。
- **性能**：拖入到归档完成目标 < 30s（不含长音频转写）。
- **成本**：每次整理是一次 Agent 调用，按量计；每日摘要 1 次/老板/天。
- **可迁移**：卸载 app 数据仍是一堆可读 Markdown，老板不被锁定。

## 13. 成功指标

- **激活**：开通后 7 天内老板有效投递 ≥ 5 次。
- **留存**：老板每周主动打开 ≥ 3 天 / 收摘要点开率。
- **代劳率**：≥ 90% 投递无需人工纠正归属。
- **节省**：老板自评"每周整理时间省下 X 小时"。

---

## 14. 风险与对策

| 风险 | 对策 |
|------|------|
| 桌面端漏掉手机高频投递 | P2 做手机"分享到同步盘"；后端是文件夹，前端可后补 |
| AI 归错公司 → 串味 | 不确定一律弹确认；归属可一键改并记忆规则 |
| 老板不信任把财务交出去 | 数据本地 + 私有同步 + 隔离铁律，作为首要卖点 |
| 同步冲突（多端编辑） | 写操作集中在 agent；老板端只读，规避并发写 |
| Tauri/Electron 选型反复 | MVP 先 Electron 抢速度，体验瓶颈再评估迁移 |

## 15. 开放问题
